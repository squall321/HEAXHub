HEAXHub Stack Showcase

Local bootstrap 9/9 healthy 21 stacks defined 14 demo integrations live 9 SIF templates packaged 2026-06-04 · v0.7.0

HEAXHub 는 사내 엔지니어링 자동화 도구들을 하나의 카탈로그로 묶고, Apptainer SIF 기반으로 격리 실행한다. 모든 통합 앱은 .portal/manifest.yaml 한 장 + 지정된 스택 컨벤션만 지키면 등록되며, stack_resolver → SIF 빌더 → 런처가 자동으로 연결한다. 아래는 현재 라이브 데모로 검증된 14개 스택의 빌드/실행 흐름.

전체 워크플로우 — Who does what

GitHub 소스 → HEAXHub 자동 빌드 → 서비스 노출까지

역할 분담 / Roles
DEV

외부 자동화 개발자

External Developer
  • 본인 GitHub 레포에 평소처럼 Flask / Streamlit / Go / Next.js 코드만 작성
  • HEAXHub 관련 파일 신경 X — .portal/ 디렉터리도 처음엔 없음
  • 그냥 git push 만 하면 됨
  • 마이너 변경(버그픽스·기능 추가)은 push → 자동 반영
OPS

HEAXHub 운영자

Operator / Reviewer
  • 신규 자동화 프로그램 등록 신청을 받음
  • HEAXHub 가 자동 생성한 manifest 초안 + run.sh 패치를 PR (또는 zip 패킷) 형태로 받음
  • 검토 후 외부 레포에 한 번만 머지
  • integrations/<slug>/.portal/manifest.yaml 에는 그 외부 레포 URL만 등록
  • 이후엔 가만히 있어도 데모는 알아서 동기화됨
SYS

HEAXHub 시스템

Automated
  • 5분마다 Celery beat 가 git pull 실행
  • 새 commit 감지 시: 매니페스트 hash 변경 → SIF 재빌드 트리거
  • 재빌드 완료 → 옛 apptainer instance stop, 새 SIF 로 start
  • Caddy 라우트 갱신 → 사용자는 거의 끊김 없이 새 버전 사용
  • 빌드 실패 시 옛 인스턴스 유지 + 운영자에게 알림

다음 단계 · 외부 자동화 프로그램 등록

엔지니어가 만든 자동화 스크립트 / 사내 서비스를 HEAXHub 에 등록하는 표준 흐름. 14개 데모가 보여준 패턴 중 하나를 골라, .portal/manifest.yaml 한 장만 추가하면 끝난다.

1스택 선택

해당 도구가 web service / dashboard / batch job / static / external 중 어디인지 결정. stacks.yaml 의 21개 중 하나를 고른다.

2manifest 작성

.portal/manifest.yaml 에 id, name, build.stack, launch.command 만 적는다. job_runner 면 inputs[] 로 폼 자동 생성.

3저장소 연결

INTEGRATION_REPO_URLS 또는 integrations/ 마운트로 등록. integrations_scanner 가 주기적으로 매니페스트를 수집한다.

4자동 빌드 · 실행

stack_resolver → SIF 빌더 → 런처가 자동 디스패치. Caddy 가 sub-path 마운트. 제출자는 카탈로그에서 카드만 클릭.
