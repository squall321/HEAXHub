SLIDE 01 · UNIFIED CONCEPT

자동화 프로그램을 "스크립트 더미"가 아니라 "앱 레지스트리"로

AI로 빠르게 만든 자동화 프로그램은 Python · JS · C++ · GUI · 웹앱 · 컨테이너가 섞이게 된다.
답은 하나의 기술로 다시 만드는 것이 아니라, Linux 허브에 manifest 기반으로 등록하고 job_id로 이력을 추적하는 것이다.

① Register (등록)
manifest.yaml로 등록
각 앱은 app_type · execution_target · owner · version을 manifest로 선언 — 구현 언어는 자유
→
② Execute (실행)
표준 인터페이스 호출
포탈은 내부 언어를 몰라도 ./run.sh input output params.json만 호출. Slurm·Apptainer·Windows Agent로 자동 라우팅.
→
③ Recall (회수)
job_id 기준 결과 통합
모든 실행은 job_id 단위로 입력·출력·로그·파라미터·버전이 보관 → 완전 재현 가능

GitHub Layer — Upstream Repo + Portal Overlay

개발자는 GitHub에서 자유롭게 만들고, 포탈 운영자는 그 위에 manifest/wrapper만 덧붙인다

2-Layer 운영
유지보수 핵심
우리 조직처럼 각자 룰 없이 만든 프로그램이 많을 때 — 먼저 GitHub에 올려 버전관리를 시작하고, 포탈 운영자가 그 위에 게시용 코드를 덧붙인다. 원본 코드는 침범하지 않는다.

신규 프로젝트는 처음부터 표준 — Template Repository 제공

새로 시작하는 자동화 프로그램은 portal-app-template repo를 "Use this template" 또는 cookiecutter로 클론하면 .portal/manifest.yaml · run.sh · README · CI 워크플로우까지 즉시 포탈 게시 가능한 형태로 시작된다.

$ gh repo create --template portal-app-template
▸ LAYER 1 · 개발자 소유
Upstream Repo
github.com/{팀}/{tool-name}
{tool-name}/ ├─ src/ # 자유로운 구현 ├─ tests/ ├─ requirements.txt ├─ README.md └─ .github/workflows/ └─ release.yml # tag push 시 사건 전파 # 개발자는 평소처럼: # git commit · PR · tag v1.2.0
→
tag 이벤트
(webhook)
▸ LAYER 2 · 포탈 운영자 소유
Portal Overlay Repo
github.com/{org}/portal-overlays/{tool-name}
{tool-name}-overlay/ ├─ .portal/ │ ├─ manifest.yaml # 포탈 등록 메타 │ ├─ run.sh # 표준 wrapper │ └─ params.schema.json ├─ upstream.lock # 핀된 버전 v1.2.0 └─ README.md # 포탈 사용 안내 # 운영자는 manifest만 관리: # upstream 코드는 절대 수정 안 함
STEP ①
개발자 tag push
git tag v1.2.0 && git push --tags
STEP ②
Upstream Release
GitHub Release 자동 생성 · webhook 발사
STEP ③
Overlay CI 트리거
overlay repo에 upstream.lock bump PR 자동 생성
STEP ④
운영자 PR 리뷰
manifest 업데이트 필요 시 함께 수정 → merge
STEP ⑤
Portal 자동 등록
app_versions 테이블에 새 버전 자동 등록 (rollback 가능)
책임 영역 개발자 (Upstream) 포탈 운영자 (Overlay)
소스 코드 구현 · 리팩토링 · 단위 테스트 읽기 전용 · 절대 수정 금지
버전 관리 git tag · CHANGELOG · semver upstream.lock으로 사용 버전 핀
manifest.yaml (템플릿 사용 시) 초안 작성 가능 최종 책임 · status · permissions · resources
run.sh wrapper core 실행 명령만 README에 명시 input/ output/ params.json 표준 어댑팅
의존성/환경 requirements.txt · Dockerfile Apptainer/SIF 빌드 · 실행 환경 배포
결과 회수 core가 결과 파일 생성 result.json · output.zip으로 표준화
사용자 문의 버그 · 기능 요청 대응 포탈 사용법 · 권한 · 이력 안내
현재
각자 PC의 스크립트, 사내 공유폴더, USB ...
버전 충돌 · 누가 만든지 모름
핫픽스 후 다른 PC는 모름
롤백 불가
▶
2-Layer GitHub
tag = 진실, overlay = 게시, 둘 다 PR로 관리
한 줄 변경도 추적 가능
핫픽스 = patch tag
롤백 = lock 되돌리기

기대 결과물

개발자는 코드에만 집중하고, 포탈 운영자는 게시 메타만 관리하며, 신규 프로젝트는 템플릿으로 즉시 표준화된다

전제 조건 · 확인 필요 항목
1. GitHub 조직 정책 — 사내 GitHub Enterprise / GitLab 사용 가능 여부 · 외부망 정책.
2. Template Repo 관리자 — portal-app-template 자체의 오너십과 업데이트 정책.
3. Overlay 운영 부담 — 앱이 많아질수록 overlay PR 처리 인원 필요 → 봇 자동화 권장.
4. 기존 앱 마이그레이션 — 운영 중인 앱을 GitHub로 옮기는 우선순위·기간 계획.

Manifest — 모든 앱은 manifest.yaml을 가진다

등록 · 실행 · 입출력 · 권한 · 리소스를 한 파일로 선언하는 단일 진실의 원천

단일 진실 원천
버전 관리
앱 이름·실행 명령·입출력·권한·리소스를 manifest.yaml 한 파일에 선언하면, 포탈은 그것만으로 등록·실행·UI 자동 생성
# LS-DYNA K File Checker — manifest 예시 id: lsdyna_kfile_checker name: LS-DYNA K File Checker version: 1.2.0 owner: CAE Automation Part status: stable # draft / beta / stable / deprecated / archived app_type: cli_tool execution_target: linux_runner launch: mode: job_runner command: ./run.sh input output params.json inputs: - name: k_file type: file required: true extensions: [".k", ".key"] outputs: - name: report_html path: output/report.html - name: summary_json path: output/result.json permissions: visibility: team # private / team / department / company executable_by: ["cae_engineer", "admin"] resources: cpu: 4 memory_gb: 8 gpu: false
🆔
식별 메타
id · name · version
owner · status · tags
⚡
실행 메타
app_type · execution_target
launch.mode · command
📥
입출력 스키마
inputs (file/number/bool)
outputs (file/folder)
🔐
권한 · 리소스
visibility · executable_by
cpu · memory · gpu

기대 결과물

포탈은 manifest를 읽는 것만으로 등록 화면 · 실행 폼 · 권한 검증 · 리소스 할당을 자동으로 처리한다

전제 조건 · 확인 필요 항목
1. 스키마 버저닝 — manifest 자체의 schema_version 명시 + 후방 호환성 정책.
2. 검증 도구 — CI에서 manifest 문법·필수필드 자동 검사 (lint).
3. 버전 스냅샷 — app_versions 테이블에 manifest_snapshot 보존하여 재현성 확보.

1. 스토리지 정책 — job 보존기간 · 대용량 결과 파일 정리(아카이브) 룰.
2. result.json 표준화 — 앱마다 다른 출력 필드를 status · summary · warnings · errors로 통일.
3. 로그 노이즈 — stdout/stderr만 모으면 너무 커지므로 app.log 구조화 권장.

AI Automation Portal 점진적 도입 로드맵

처음부터 완전 통합을 목표로 하지 않는다. App Registry → Job History → 결과 표준화 → Windows 연동 → Core 엔진화의 5단계로 나누어 단계적으로 확장한다.
각 단계는 다음 단계의 전제이며, 1·2단계만으로도 즉시 가치를 제공한다.

핵심 결론 · "통일하지 말고, 등록과 회수만 통일하라"

혼합 앱 환경에서 정답은 "하나의 기술로 다시 만드는 것"이 아니다.
앱의 구현 방식은 그대로 둔다. 대신 등록 방식 · 실행 방식 · 결과 회수 방식 · 이력 관리 방식을 표준화한다.
그 출발점은 GitHub Upstream + Portal Overlay의 2-Layer 운영이며, 신규 프로젝트는 Template Repo로 즉시 표준화한다.

Linux 허브를 중앙 포탈로 두고 Windows GUI · 웹앱 · CLI · Slurm · Apptainer 앱을 함께 관리하면, AI로 빠르게 만들어진 다양한 자동화 프로그램이 단순한 스크립트 더미가 아니라, 조직적으로 관리 가능한 Automation Tool Registry / Internal App Portal로 발전한다.
