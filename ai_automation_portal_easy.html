AI 자동화 포탈
흩어진 프로그램을 한 곳에서
팀마다, 사람마다 만든 자동화 프로그램이 너무 많고 흩어져 있어요.
이걸 전부 다시 만들지 않고도, 한 곳에서 찾아 쓰고 결과를 모을 수 있게 만드는 안내서입니다.
이 문서를 어떻게 읽으면 되나요?  각 페이지 위쪽에 파란색 "이게 뭐예요?" 박스가 있어요. 어려운 단어가 나오면 그 박스를 먼저 보세요 — 일상 비유로 풀어서 설명합니다.
포탈 (관리하는 곳) — 등록·실행·이력 규칙
자동화 프로그램들 — 지금 형태 그대로 유지
지금 쓰는 프로그램은 그대로 둡니다.
대신 포탈에 "이렇게 부르면 실행돼요" 한 줄만 등록하면 됩니다.
포탈은 모든 걸 새로 만드는 곳이 아니라, 한곳에 모아 안내하는 곳입니다.

이게 뭐예요?
한 줄로 말하면 — "앱 스토어" 같은 사내 사이트를 만드는 이야기예요
스마트폰의 앱 스토어에서 우리가 앱을 찾고, 누르고, 설치하잖아요? 그런데 회사 안에 흩어진 자동화 프로그램들도 똑같이 한곳에서 찾아 쓰고 결과를 받을 수 있게 하자는 겁니다.
앱 스토어가 앱을 직접 만들지 않듯, 우리 포탈도 프로그램을 새로 만들지 않아요. 그저 "여기 이런 프로그램이 있어요, 이렇게 쓰세요" 라고 안내하고 결과를 모아주는 역할입니다.
슬라이드 01 · 전체 개념

흩어진 프로그램을 "한 곳에서 찾아 쓰는" 사내 앱 스토어로

AI로 빠르게 만든 자동화 프로그램은 종류도 만든 방식도 다 달라요.
전부 다시 만들 필요 없습니다. 한 곳(포탈)에 "이런 프로그램이 있어요" 라고 등록만 하고, 누가 언제 무엇을 실행했는지 작업 번호(job_id)로 기록해 두면 충분합니다.

① 등록하기
"이런 프로그램이에요" 명함 만들기
프로그램마다 이름·만든 사람·버전·실행 방법을 적은 작은 안내 카드(manifest)를 만들어 포탈에 붙입니다.
→
② 실행하기
버튼 한 번으로 실행
사용자는 그저 실행 버튼만 누릅니다. 포탈이 알아서 "이 프로그램은 어디서 돌릴지" 정하고 실행시켜요.
→
③ 결과 받기
작업 번호로 결과 보관
한 번 돌릴 때마다 작업 번호(job_id)가 생겨요. 입력 파일·실행 옵션·결과가 그 번호 안에 다 모여 — 6개월 뒤에도 똑같이 다시 돌릴 수 있어요.
▶ 전체 그림 — 중앙 포탈이 여러 종류의 실행 장소를 연결합니다
// Linux Portal Server (중앙 허브)
portal_web: React + TS + Ant Design
backend_api: FastAPI / Flask
registry_db: PostgreSQL · apps · versions
job_history: jobs · permissions · logs
file_storage: NAS / NFS / MinIO
queue: Redis Queue · Celery
↓ dispatch by execution_target
backends: linux · apptainer · slurm · windows_agent
"모두 같은 모양으로 만들지 말고, 부르는 방법만 똑같이"
지금 쓰는 프로그램을 그대로 두고,
포탈은 "명함"으로 찾아서, "작업 번호"로 결과를 모을 뿐입니다.
심지어 Windows 프로그램도 작은 도우미를 통해 같은 흐름으로 들어옵니다.

코드 보관소 두 층 — 개발자와 운영자가 따로 일해요

개발자는 평소처럼 자유롭게 코드를 만들고, 포탈 운영자는 그 위에 "포탈에 게시할 안내"만 덧붙입니다

2층 구조
관리가 쉬워요
이게 뭐예요?
GitHub은 "코드를 보관·공유하는 사물함"이에요
같은 문서를 여러 사람이 같이 수정하면 누가 뭘 바꿨는지 헷갈리잖아요? GitHub은 그걸 줄 단위로 기록·복원해 주는 코드용 사물함입니다. 우리는 그 사물함을 두 칸으로 나눠서 쓰자는 이야기입니다.
1층(개발자 사물함) — 요리사가 쓰는 주방. 자기 레시피·재료를 자유롭게 둠.
2층(포탈 사물함) — 가게 메뉴판. "어떤 요리를, 얼마에, 어떻게 주문" 만 적혀 있음.
주방을 안 건드리고 메뉴판만 고치면 되니, 유지보수가 훨씬 쉬워져요.
우리 조직처럼 각자 룰 없이 만든 프로그램이 많을 때 — 먼저 GitHub에 올려 버전관리를 시작하고, 포탈 운영자가 그 위에 게시용 코드를 덧붙인다. 원본 코드는 침범하지 않는다.

새로 시작하는 프로젝트는 — "기본 양식" 한 벌 제공

새 프로그램을 만들 때는 빈 종이부터 시작하지 마세요. 회사가 미리 만들어 둔 기본 양식을 복사해서 시작하면, 포탈에 바로 등록될 수 있는 형태로 자동으로 갖춰져요. — 마치 이력서를 빈 워드로 쓰지 않고 양식 파일을 받아 채우는 것과 같습니다.

"이 양식으로 시작" 버튼 클릭
▸ LAYER 1 · 개발자 소유

→
tag 이벤트
(webhook)
▸ LAYER 2 · 포탈 운영자 소유
Portal Overlay Repo
github.com/{org}/portal-overlays/{tool-name}
{tool-name}-overlay/ ├─ .portal/ │ ├─ manifest.yaml # 포탈 등록 메타 │ ├─ run.sh # 표준 wrapper │ └─ params.schema.json ├─ upstream.lock # 핀된 버전 v1.2.0 └─ README.md # 포탈 사용 안내 # 운영자는 manifest만 관리: # upstream 코드는 절대 수정 안 함

개발자가 새 버전 발표

단계 ②
자동 알림
새 버전이 나왔다는 신호가 운영자에게 자동 전달
단계 ③
운영자 사물함에 자동 메모
"새 버전 쓸까요?" 라는 검토 요청 자동 작성
단계 ④
운영자 확인
새 버전을 살펴보고 OK / 보류 결정
단계 ⑤
포탈에 자동 게시
사용자가 포탈에서 새 버전을 바로 쓸 수 있음
맡은 일 개발자 (코드 만든 사람) 포탈 운영자 (게시 담당)
프로그램 코드 새로 만들고 · 고치고 · 시험함 읽기만 · 절대 손대지 않음
버전 관리 "이제 1.2 버전" 같이 깃발 꽂기 "우리는 1.2 버전 사용 중" 메모만
앱 명함 (양식 쓸 때) 초안만 살짝 작성 최종 책임 — 누가 쓸 수 있나, 얼마나 큰 일인가
실행 방법 "이렇게 켜는 거예요" 사용설명서 포탈이 부를 수 있게 한 줄로 정리
필요한 환경 "이 도구 필요해요" 목록 작성 실제로 그 환경을 서버에 설치·준비
결과물 프로그램이 알아서 결과 파일 만듦 요약·전체파일을 통일된 형태로 정리
사용자 문의 "버그예요 / 기능 추가해주세요" "포탈에서 어떻게 써요?" "권한 문의"

개발자는 코드에만 집중, 운영자는 게시 안내에만 집중. 새 프로젝트는 양식으로 즉시 시작!

그 전에 확인할 일들
1. 어디에 사물함을 둘까? — 사내 GitHub 쓸지 외부망 쓸지 정책 확인.
2. 양식은 누가 관리할까? — 기본 양식 자체를 책임지고 업데이트할 담당자.
3. 운영자 일이 너무 많아지지 않을까? — 앱이 늘어나면 자동 봇으로 검토 부담 줄이기.
4. 기존 프로그램은 언제 옮길까? — 중요한 것부터, 시간 여유를 두고 천천히.

프로그램 종류 — 7가지로 나누어요

모든 프로그램은 "어떤 종류인가요?" 한 줄로 분류됩니다 — 그래야 포탈이 알맞게 안내할 수 있어요

7가지 분류
모든 앱이 가짐
이게 뭐예요?
프로그램을 종류별로 라벨링하는 거예요
도서관 책에 "소설/과학/만화" 라벨이 붙어 있듯, 포탈의 모든 프로그램에도 "이건 명령어형이에요" "이건 웹사이트예요" "이건 윈도우 프로그램이에요" 라는 라벨을 붙여요. 그래야 포탈이 알맞게 다룰 수 있어요.
식당 메뉴판이 "한식 · 양식 · 분식 · 디저트"로 나뉘어 있는 것과 같아요. 라벨이 있어야 손님이 찾기 쉽고, 주방도 어디로 갈지 알아요.
윈도우 프로그램이든, 웹사이트든, 명령어 도구든 — "종류" 라벨 한 줄만 붙이면 모두 포탈에서 똑같이 다뤄집니다

실행되는 장소 — 이 프로그램은 어디서 돌아가요?

같은 프로그램도 "내 PC에서 / 사내 서버에서 / 윈도우 서버에서" 등 돌아가는 곳이 달라요

자동 분배
사용자는 몰라도 OK
이게 뭐예요?
"이 일은 어느 부서에 맡길까?" 같은 거예요
회사에서 어떤 일이 들어오면 누군가가 "이건 회계팀 일, 저건 디자인팀 일" 이라고 정리해주잖아요? 포탈도 똑같이 "이 프로그램은 어디서 돌리면 되는지" 정해서 자동으로 보내줘요. 사용자는 그저 실행 버튼만 누르면 됩니다.
택배 분류센터를 생각해 보세요. 우리는 상자만 맡기고, 분류센터가 알아서 "이건 서울행, 저건 부산행" 정해줘요. 포탈도 그런 역할입니다.

이게 뭐예요?
윈도우 프로그램을 "심부름꾼"이 대신 실행하는 방법이에요

슬라이드 08 · 단계별 도입 + 10가지 약속

AI 자동화 포탈 단계별 도입 계획

처음부터 모든 걸 하지 않습니다. GitHub 이주 → 카탈로그 → 작업 기록 → 결과 통일 → 윈도우 합치기 → 핵심 분리 순서로 단계적으로 확장합니다.

# 단계 하는 일 그 전에 필요한 것 시점
0 GitHub 사물함 + 기본 양식 기존 프로그램을 GitHub에 모으기 · 새 프로젝트용 양식 배포 GitHub 조직 정책 · 옮길 순서 결정 토대 (가장 먼저)

핵심 한 줄 · "프로그램은 그대로 두고, 부르는 방법만 통일하자"

여러 종류의 자동화 프로그램이 섞여 있을 때 정답은 "하나로 다시 만드는 것"이 아닙니다.
프로그램은 지금 모습 그대로 두고, 등록·실행·결과 회수·기록 방식만 같이 맞춥니다.
그 시작은 GitHub에 모으는 것이고, 새 프로젝트는 기본 양식으로 즉시 시작합니다.

이렇게 하면, AI로 빠르게 만든 흩어진 자동화 프로그램들이 "누구 PC에 있는지 모를 스크립트 더미"가 아니라, 한 곳에서 찾아 쓰고 결과를 모을 수 있는 회사의 자산이 됩니다.
